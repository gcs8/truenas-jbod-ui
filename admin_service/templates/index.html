      Guided Walkthrough

      Create Or Revise A System

      Start fresh for a new host or load an existing saved system into the form, preview the profile you are about to pin, and prep or reuse an SSH key before saving back into the shared config.

      1

      Basics

      Name the system, pick a platform, and decide whether to pin a default enclosure profile.

      2

      API And Scope

      Point the new system at the right host and auth path before you decide how much SSH enrichment to add.

      Uses normal CA and hostname validation from the sidecar trust store. This works with public CAs and private CAs you install into the container, but it does not do certificate-thumbprint pinning yet.

      Use this when you connect to an IP or alternate address, but the certificate is issued to a DNS name.

      BMC

      BMC / IPMI Access

      Use this for Supermicro-style out-of-band inventory, drive locate control, and node UID control. It can stand on its own for BMC-only systems or enrich ESXi and other host-managed platforms later.

      Supermicro BMC support prefers Redfish when the enclosure endpoints are available, then falls back to the validated web XML path for storage locate and UID control.

      Quantastor HA Nodes

      Reuse the shared SSH auth above, then point each Quantastor HA node at its own host so internal views like SATADOMs can target the correct node cleanly.

      Optional

      One-Time Bootstrap

      Use temporary root or installer credentials once to create the final service account, install the selected public key, and optionally write limited sudo rules.

      These bootstrap credentials are used for this action only. They are never written into config.yaml. Leave the saved SSH password fields above blank if you want the final service account to stay key-only.

      Bootstrap is off by default for saved systems. Enable it only when you intend to run one-time service-account setup.
